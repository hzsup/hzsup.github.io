Git基础

概念

Git是分布式版本控制工具，用于管理项目中文件的修改、更新历史，可以做到历史记录查看、版本回溯、多人协同开发。

版本控制

通常情况下，软件项目涉及多人协作，对于同一个项目，大家都在项目中进行添加文件、修改文件、删除文件等操作，因此在开发中需要一个工具来进行管理，比如当前文件这部分被谁修改过、当前项目出现问题如何快速回滚到上一个版本、当前文件同一个部分被两个人修改到底以谁的结果为准。版本控制工具就是为了解决这个问题。

纯手动版本控制

以毕业论文为例，最初的毕业论文的文件名为毕业论文。对毕业论文改了一版，命名毕业论文02，之后会出现毕业论文03、毕业论文04、毕业论文最终版01等等。这种就是手动版本控制。
优点：可以快速回滚。
缺点十分明显：

  1. 无法进行多人协作开发
  2. 无法显式看到文件不同版本之内内容的区别
  3. 存储空间浪费严重，每一版可能只修改不到10%内容，但是进行全量存储

diff与patch

diff和patch是版本控制中两个重要的工具。
diff -u filename01.c filename02.c可以比较文件filename01.c和filename02.c两个文件的差异。命令会输出一个diff.txt文件，记录两个文件的不同。
patch用于更新或者恢复文件。比如patch filename01.c diff.txt可以把文件filename01.c的内容更新，更新之后内容和filname02.c相同。patch -R filename02.c diff.txt可以反向执行，将filename02.c文件内容回滚到filename01.c。
有了这两个工具，可以做到：

  1. 查询一个文件的历史修改记录
  2. 文件可以快速回滚到之前版本

本地版本控制工具RCS

借助diff和patch可以搭建本地版本控制工具，记录文件的初始状态，每次更新通过diff命令记录更新内容文件，把这些diff.txt文件集合保存，得到本地版本控制工具RCS。
优点：

  1. 可以查询文件历史修改记录
  2. 可以快速回滚文件到历史版本
    缺点： 无法多人协同开发。

集中式远程版本控制工具CVS、SVN

对于企业复杂项目，必须支持多人远程协同开发。
项目文件在远程服务器存放，项目团队开发者们从服务器获取项目文件，本地开发之后提交到服务器，就是远程版本控制工具CVS。
CVS存在一个致命缺点：不支持原子提交。比如开发者本地修改了3个文件，提交到服务器，可能由于网络或者其他原因，3个文件中2个修改成功1个修改失败。并且这个操作是无法回滚的，需要人工手动排查。这个是实际开发中是不能接受的。
SVN修复了这个问题，实现了原子提交，优化了服务器内容存储，是一个可用的版本控制工具。
但是SVN存在一些缺点，也是软件采用集中式架构通用的缺点。

  1. 效率低下，操作延迟高：所有的项目开发者通过服务器的操作太频繁，查看日志、提交代码。导致很多操作需要排队，服务器压力很大
  2. 单点故障：一旦服务器出问题或者遭受黑客攻击，开发直接停摆
  3. 存储链路脆弱：采用原始文件加diff集合的方式存储，如果原始文件某一部分出错或者某一个diff文件缺失等问题，链路无法正常进行，后续文件无法恢复查看

分布式版本控制工具Git

上述问题由于集中式架构带来的，Git采用了分布式架构，在保证功能可用情况下，大大提升了工具的效率以及健壮性。
架构设计上，Git架构包括远程仓库（服务器）和本地仓库部分。开发者在开发时将文件更新，修改等操作提交到本地仓库，可以从本地仓库查看文件历史记录等，而服务器主要进行同步，比如从服务器获取最新项目代码、本地仓库进度同步到服务器。这样的操作好处：

  1. 减少与服务器交互，服务器压力小：之前所有操作都需要通过服务器，现在查看历史记录可以在本地，修改文件等操作可以提交到本地仓库，攒一块提交到服务器。
  2. 健壮性强：即使服务器出现故障，不影响开发者操作。服务器数据丢失，每个开发者本地仓库保留数据。
  3. 记录快照的数据存储方式：相比如记录文件的变更diff列表，Git每次提交更新记录当前全部文件的快照，如果文件没有被修改，保留链接指向上一个文件

Git配置

Git配置分为三个等级：系统配置、用户配置、项目配置。
系统配置的范围是当前系统所有用户，用户配置的范围是当前系统中的某一用户，项目配置范围是对应当前项目。
Git加载配置时，首先加载系统配置，然后加载用户配置，如果用户配置属性和系统配置不一样，覆盖系统配置。最后加载当前项目配置，同样如果不一样覆盖之前的配置。

常见配置

配置用户名与邮箱，用于项目开始时提交修改记录的信息记录，适合放到用户配置下。

windows系统下文本换行符为CRLF,而Linux和Mac下文本换行符为LF。如果涉及跨平台开发，可能会出问题。
首先仓库的文件的文本换行符以LF为准，在windows可以打开配置，提交文件时自动把CRLF转换为LF，从仓库获取文件时将LF转换为CRLF。

Git结构

Git是分布式版本控制工具，分为四个部分：远程仓库、本地仓库、暂存区、工作区。远程仓库就是服务器，本地仓库、暂存区、工作区都是在本机上。
工作区：项目开发中项目文件、代码都属于工作区。
暂存区：位于工作区和本地仓库之间的中间层，文件修改后提交到暂存区。
本地仓库：项目.git目录下，用于管理工程的版本数据，暂存区保存的文件可以提交到本地仓库。
Git常见流程：

  1. 通常来说，我们首先打开项目文件，进行代码开发，开发完成之后保存文件，这时候文件处于已修改的状态。
  2. 通过git add命令，可以将已经修改的文件添加到暂存区，暂存区的文件是已暂存。可以继续进行开发工作。
  3. 等完成当前需求功能的几个文件并全部通过git add添加到暂存区之后，git commit可以提交暂存区文件到本地仓库，此时文件状态变为已提交。
  4. 之后可以通过git push将代码推送到远程仓库中，这样别人也可以看到我写的代码。

Git命令

git init 项目名可以在本地创建一个Git管理的项目，项目下.git目录是管理工程的版本数据。
git clone 项目地址可以从远程仓库拉取一个Git项目。
git add 文件名 添加文件到暂存区。分为两种情况：

  1. 当前文件新建，之前项目没有，通过git add添加到暂存区进入git跟踪
  2. 如果文件存在已经被git管理，只是修改了部分内容，通过git add将文件的最新快照保存到暂存区
    git diff可以比较项目中两个分支的差异，比较工作区和暂存区的差异等。通过--name-status参数可以只看文件列表。
    git status可以查看工作目录和暂存区状态，包括哪些文件未纳入git跟踪，哪些文件在暂存区未提交，哪些被git管理文件修改了但是未纳入暂存区。
    git commit -m "描述"将暂存区文件提交到本地仓库。
    git log查看项目的提交日志。
    git push 远程仓库名 本地分支名:远程分支名把本地仓库的代码推送到远程仓库，这样别人也可以看到。如果本地分支名和远程分支名一样，直接写一个分支名就行。
    git branch查看项目本地分支，-a参数可以查看本地和远程全部分支。
    git branch 分支名创建分支但是不切换到新分支。
    git checkout -b 分支名创建分支并且切换到新分支。
    git branch -d 分支名删除分支，-D可以强制删除分支（如果分支上存在未合并的改动使用-d无法删除分支）。
    git checkout 分支名切换到其他分支。
    git fetch 远端仓库名称 分支名从远端服务器获取最新的分支，不会进行合并，不影响当前分支。
    git pull 远端仓库名称 远端分支名:本地分支名从远端服务器获取最新的分支，并于本地的分支进行合并。如果分支名一样写一个就行。git pull等于git fecth+git merge。
    git merge 分支名将分支合并到当前所在的分支。
    git rebase 分支名合并分支，将当前分支的最新提交搬到目标分支的最新节点之后。合并方式和merge不同。
